JOB PORTAL EC2 & DOCKER DEPLOYMENT GUIDE
========================================

Since your application uses a microservices architecture (5 backend services + 1 Next.js frontend) and requires Kafka, deploying to a single EC2 instance requires careful resource management and Docker orchestration.

This guide outlines how to containerize everything using `docker-compose` and host it on a single AWS EC2 instance.

---

### STEP 1: PROVISION THE AWS EC2 INSTANCE
A total of ~6 Node.js applications plus a Kafka/Zookeeper cluster will consume significant memory. Do NOT use a `t2.micro` (1GB RAM) as it will crash.

1. Go to your AWS Management Console -> EC2 -> Launch Instance.
2. OS Image: Select "Ubuntu Server 22.04 LTS" or "24.04 LTS".
3. Instance Type: Select `t3.medium` (4GB RAM) or `t3.large` (8GB RAM - Highly Recommended).
4. Key Pair: Create a new key pair (.pem) or select an existing one to SSH into your instance.
5. Storage: Allocate at least 20-30 GB of gp3 SSD.
6. Network Settings (Security Group): Allow the following inbound traffic:
   - SSH (Port 22) from Anywhere (or your IP)
   - HTTP (Port 80) / HTTPS (Port 443)
   - Custom TCP (Port 3000) for Frontend
   - Custom TCP (Ports 5001 - 5005) for Backend Services 
     *(Note: If configuring an NGINX reverse proxy on port 80 later, you won't need to open 5001-5005 publically).*

---

### STEP 2: INSTALL DOCKER & DOCKER-COMPOSE ON EC2
Connect to your instance via SSH:
`ssh -i "your-key.pem" ubuntu@<your-ec2-public-ip>`

Run the following commands to install Docker:
```bash
sudo apt-get update -y
sudo apt-get install ca-certificates curl gnupg -y
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg

# Add the repository to Apt sources:
echo \
  "deb [arch="$(dpkg --print-architecture)" signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu \
  "$(. /etc/os-release && echo "$VERSION_CODENAME")" stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

sudo apt-get update -y
sudo apt-get install docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin docker-compose -y

# Add user to docker group (so you don't need 'sudo docker')
sudo usermod -aG docker $USER
# Log out and log back in to apply group changes
```

---

### STEP 3: DOCKERIZE YOUR SERVICES
Before pulling code to EC2, ensure the project is fully Dockerized in your repository.

1. You are already creating `Dockerfile` and `.dockerignore` for each service. A standard Node.js Dockerfile for your backend services looks like:
```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build
EXPOSE 5001 # (Change port dynamically per service)
CMD ["npm", "start"]
```

2. Your Next.js Frontend `Dockerfile`:
```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build
EXPOSE 3000
CMD ["npm", "start"]
```

---

### STEP 4: CREATE DOCKER-COMPOSE
In the root directory of your project (where `frontend` and `services` folders sit), create a `docker-compose.yml`. This orchestrates all 6 services + Kafka + Zookeeper so they can communicate together seamlessly.

```yaml
version: '3.8'

services:
  zookeeper:
    image: confluentinc/cp-zookeeper:latest
    environment:
      ZOOKEEPER_CLIENT_PORT: 2181
      ZOOKEEPER_TICK_TIME: 2000

  kafka:
    image: confluentinc/cp-kafka:latest
    depends_on:
      - zookeeper
    ports:
      - "9092:9092"
    environment:
      KAFKA_BROKER_ID: 1
      KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:29092,PLAINTEXT_HOST://localhost:9092
      KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: PLAINTEXT:PLAINTEXT,PLAINTEXT_HOST:PLAINTEXT
      KAFKA_INTER_BROKER_LISTENER_NAME: PLAINTEXT
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1

  auth-service:
    build: ./services/auth
    ports:
      - "5001:5001"
    env_file: ./services/auth/.env
    depends_on:
      - kafka

  utils-service:
    build: ./services/utils
    ports:
      - "5002:5002"
    env_file: ./services/utils/.env
    depends_on:
      - kafka

  user-service:
    build: ./services/user
    ports:
      - "5003:5003"
    env_file: ./services/user/.env

  job-service:
    build: ./services/job
    ports:
      - "5004:5004"
    env_file: ./services/job/.env
    depends_on:
      - kafka

  payment-service:
    build: ./services/payment
    ports:
      - "5005:5005"
    env_file: ./services/payment/.env

  frontend:
    build: ./frontend
    ports:
      - "3000:3000"
    env_file: ./frontend/.env
    depends_on:
      - auth-service
      - user-service
      - job-service
```

*Important Detail:* 
Because all services are running in Docker, inside your `.env` files replace `KAFKA_BROKER=localhost:9092` with `KAFKA_BROKER=kafka:29092`. (Docker containers talk to each other using internal network DNS names).

---

### STEP 5: DEPLOYMENT ON EC2
1. Clone your project onto the EC2 instance.
`git clone <your-repo-url>`
`cd Job_portal`

2. VERY IMPORTANT: `.env` files are not pushed to Github. You must manually recreate your `.env` files on the EC2 instance inside every service folder and the frontend using nano/vim. 
`nano services/auth/.env`

3. Update the frontend's API Endpoints.
Instead of `localhost`, your frontend's `api.ts` needs to be aware of the public IP of your EC2 instance so that the browser can route requests properly. 
In your `frontend/.env`, set:
```env
NEXT_PUBLIC_AUTH_URL=http://<YOUR_EC2_PUBLIC_IP>:5001/api/auth
NEXT_PUBLIC_USER_URL=http://<YOUR_EC2_PUBLIC_IP>:5003/api/user
NEXT_PUBLIC_JOB_URL=http://<YOUR_EC2_PUBLIC_IP>:5004/api/job
NEXT_PUBLIC_PAYMENT_URL=http://<YOUR_EC2_PUBLIC_IP>:5005/api/payment
```

4. Build and Run the Stack:
```bash
docker-compose up -d --build
```
`--build` ensures everything has an image created.
`-d` runs it in "detached mode" in the background so it stays alive when you disconnect SSH.

5. Check the status of your microservices:
```bash
docker-compose ps
# To check logs for a specific service that might have crashed:
docker-compose logs -f auth-service
```

### PRO-TIPS FOR PRODUCTION
- NGINX Reverse Proxy: Instead of opening ports 5001 through 5005 to the public internet, deploy an Nginx container. Have Nginx listen on Port 80, and reverse proxy traffic routing based on the path (eg. /api/auth -> auth-service:5001). This secures your backend ports from the public internet.
- SSL/TLS (HTTPS): Once you map a domain name to your EC2 instance, use Certbot to quickly bind an SSL certificate through Nginx so your users have a secure (HTTPS) connection.
